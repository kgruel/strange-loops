# Addendum: Multi-Modal Rendering & The "Broadcast" Pattern

**Context:** Exploring the architectural capability of `ev` to support simultaneous, diverse outputs (UI, Log, DB, Web) via the `TeeEmitter` pattern.

## The Power of "Teeing"

Because `ev` decouples **Emission** from **Handling**, you can broadcast the same event stream to multiple "Sinks" with radically different purposes.

### 1. The Multi-Channel User Experience

The scenario you described—"Beautiful CLI view + Structured Log File for LLM"—is the canonical use case.

```python
# Conceptual wiring in ev-toolkit / hlab
emitter = TeeEmitter(
    # Channel 1: The Human (Rich UI)
    RichEmitter(theme=DraculaTheme),
    
    # Channel 2: The Robot (JSONL)
    FileEmitter("debug.jsonl", format=JsonFormat),
    
    # Channel 3: The System (Structlog Bridge)
    StructlogBridgeEmitter(),
)
```

**Why this is special:**
Most CLIs force you to choose: `mytool --json` OR `mytool --pretty`.
`ev` allows `mytool` to be **both** simultaneously.
*   **Stdout:** Pretty UI (ephemeral).
*   **File:** Structured Data (durable).

### 2. Beyond Files: The "Live Stream" Opportunity

Since `ev.Event` is just a serializable dict, you can pipe it over the network easily.

#### A. The "Web Dashboard" Emitter
Imagine a `WebsocketEmitter`.
*   **Action:** Pushes events to a lightweight server (`fastapi` + `htmx`?).
*   **Result:** A colleague can watch your deployment progress **live in their browser** just by visiting a URL, while you watch it in your terminal.
*   **Mechanism:** `ev-present` Atoms (`StateLine`, `WorkLine`) map cleanly to HTML components (`<div class="stateline">`, `<progress>`).

#### B. The "SQLite Audit" Emitter
Imagine `SqliteEmitter`.
*   **Action:** Inserts events into a local `.db` file.
*   **Use Case:** "Queryable History".
    *   `SELECT avg(value) FROM events WHERE kind='metric' AND name='duration'`
    *   Review past deployments with SQL.

#### C. The "LLM Context" Stream
This is your specific use case.
*   **Scenario:** A deployment fails.
*   **Action:** You feed the `debug.jsonl` file (generated by the Tee) into an LLM.
*   **Benefit:** Because `ev` events are **Semantic** ("Step 'Decrypt' started", "Artifact 'config' created"), the LLM understands the *intent* of the operation much better than parsing raw text logs ("grep: file not found").

## 3. The "Headless" Advantage

Because `ev-present` defines **Atoms** (`StateLine`, `WorkLine`) independently of `rich`, these alternative renderers don't have to reinvent the wheel.

*   **Rich Backend:** `StateLine(state="error")` → `[red]✗[/]`
*   **HTML Backend:** `StateLine(state="error")` → `<span class="icon-error">✗</span>`
*   **Slack Backend:** `StateLine(state="error")` → `:x:`

The **Adapter Layer** (Event → Atom) is shared.
The **Backend Layer** (Atom → Output) is swappable.

## Conclusion

The `ev` architecture accidentally solved the "Observability vs. UX" trade-off. By treating the User Interface as just "one of many subscribers" to the event stream, you gain the ability to add **Audit**, **Remote Viewing**, and **AI Context** channels effectively for free.
