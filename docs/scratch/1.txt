What is allowed to be a fact?

Can facts be revoked, contradicted, superseded, or only countered?
Can a fact be malformed but still admitted?

A shared fact-space needs a theory of admissibility, not just immutability.

So where does this break?

It breaks where the model has to stop being an ontology and start making irreversible operational choices.
Not “breaks” as in collapses. More like: these are the stress fractures where you need explicit policy, invariants, or scope cuts.

1. It breaks at bad admission
If fact-space admits everything, then everything becomes operationally real somewhere. Horizon and Potential limit consequence, but the garbage still exists.
That is fine if fact-space is a ledger of “claims made.” It is dangerous if readers treat fact-space as “things that are true.”

So the invariant has to be:

Fact-space admits assertions, not truth.
A fact is not “the world is this way.”
A fact is “this observer emitted this assertion from this position at this time.”
If you keep that discipline, bad facts are survivable. If not, the system becomes a truth-laundering machine.

2. It breaks when Horizon becomes too dynamic
Horizon is powerful because it bounds what an observer can see.
But if facts can derive changes to Horizon, then the system can accidentally create privilege escalation loops:

observer emits fact
derivation interprets fact
tick changes observer horizon
observer now sees more
observer emits stronger facts
That may be intended for learning agents. It is also exactly where things get dangerous.
So you need a hard distinction between:

facts that describe horizon
facts that propose horizon
facts that actually grant horizon
Otherwise “the system learned I should see X” becomes indistinguishable from “I am now allowed to see X.”
Keeper line:

Derived horizon is not granted horizon.
3. It breaks when Potential becomes authority laundering
Same problem, but with action.
If an untrusted observer can emit a fact that participates in a derivation that grants it more Potential, you have a self-amplifying actor.

That can be useful for adaptive agents, but only if there is a separate authority loop whose Potential includes granting Potential.

So:

Potential can only be expanded by an observer-position that already has grant-potential.
This is probably a core safety invariant.
Otherwise the strange loop becomes a permission ouroboros.

4. It breaks when fact equality is mistaken for evidential equality
This one is subtle and likely.
Observations and ticks are both facts. Good.

But if downstream folds forget that a tick is a heavy derived fact and treat it like a fresh observation, you get compounding distortion.

Example:

raw fact: user clicked thing once
tick: user is interested in thing
later tick: user strongly prefers thing
later tick: always show user thing
Each level may be fact-shaped, but the evidential quality is changing. If folds do not account for depth, density, and provenance, the system can bootstrap weak claims into strong ones.
So:

Fact equality is structural equality, not evidential equality.
And:
Depth must be visible to derivations that make consequential claims.
5. It breaks when weight becomes ambiguous
“Weight” is doing too much unless you split it.
Weight might mean support count, derivation depth, confidence, authority, recency, semantic importance, or operational consequence.

Those cannot all be one number.

If you keep “weight” as a poetic super-term, fine. But code needs separable fields or typed measures.

Maybe:

density = how many facts folded in
depth = how many derivation collapses are inside
confidence = how strongly the derivation claims its output
authority = who/what is allowed to make this claim matter
impact = what this fact can affect
If weight is not decomposed, the model invites accidental math:
“This fact is heavier, therefore it matters more.”
That is not necessarily true.
A fact can be dense, deep, low-confidence, high-authority, stale, and dangerous all at once.

6. It breaks at lossy collapse
Collapse is elegant: derivation product re-enters fact-space without changing type.
But collapse is also compression. If the tick does not preserve enough references, inputs, derivation metadata, and context, then the system cannot actually recover what was folded.

At that point, “levels are recoverable traces” becomes false.

So the hard invariant is:

A tick without sufficient references is not unfoldable; it is only asserted.
That may be allowed. But it must be explicit.
You may need classes:

unfoldable tick
auditable tick
opaque tick
Not every tick can or should be fully replayable. But the model must not pretend all are.

Earlier we said references are optional to consult. That is true for performance and projection.

References may be optional to read; they are not optional to carry for derived ticks that claim weight.

8. It breaks with cycles of self-description
The observer loop is where the system becomes properly strange, but it is also where it can become unstable.
If facts about the observer shape the observer, and the observer emits new facts about itself, you can get identity drift, self-confirmation, or hallucinated self-state.

memory fold emits: agent is good at X
prompt fold includes: you are good at X
agent emits more confident X claims
That is a strange loop. It is also a bias amplifier.

Describing the observer is cheap. Reconstituting the observer is privileged.

A system that can represent everything can become uselessly general.

That phrase matters.

Append-only is clean until you need privacy, retention, legal deletion, or “this fact should not be used anymore.”

That is probably the central architectural split.
Canonical:

Fact-space is permissive.
Vertex-space is operationally bounded.
Or:
Facts record concern; vertices govern consequence.
That is a keeper.
Because it answers nearly every safety/trust question:

bad fact? It can be recorded.
untrusted observer? Their facts exist.
can it matter everywhere? No, because vertices have Horizon.
can it do anything? No, because vertices have Potential.
can it become stronger by citation? Only as density/depth, not truth.
can policy mutate itself? Not unless vertex topology explicitly allows it.
The remaining genuine breakpoints
With your clarifications, I’d reduce the serious breakpoints to five:
1. Misreading facts as truth
You already know this. It is the largest docs hazard.
The line should probably be unavoidable:

A fact is not truth. A fact is an attributed record of concern.
Maybe even:
Facts are not true things. Facts are recorded noticings.
2. Misreading weight as authority
This is the second biggest.
Weight is reference density, not confidence.
If people miss that, they will build reputation engines accidentally.
3. Under-specifying observer identity
Your #15 remains worth thinking about because “observer cared to emit it” makes observer attribution central.
You probably want a very crisp observer model, even if minimal:

observer_id
emitted_at
vertex_id maybe?
kind/source?
payload
references
The thing to avoid is “observer” becoming a vague label when it is actually the source of fact admission.
4. Explaining why this is not just an append-only event store with references
This is not a fatal critique, but it is the skeptical doorway.
The answer might be:

It is an append-only event store when projected as storage. It becomes Strange Loops at the point where ticks are facts, loops meet at horizon/potential-bounded vertices, and reference depth is recoverable as levels.
That keeps “projections, not equivalences.”
5. Debuggability / explanation
If vertices define Horizon/Potential, then explaining a downstream tick requires:
why these facts were visible at this vertex
why this vertex had potential to emit this tick
which references were included
which facts were excluded or invisible
what later vertices consumed it
This is not a model flaw. It is the primary UX burden.

One canonical block I would emit now
A fact is an attributed record of concern: an observer cared to emit it. Fact-space admits facts; it does not certify them as true.

Vertices govern consequence. Each vertex has a horizon, which bounds what it can see, and a potential, which bounds what it can emit or do. Loops meet at vertices; ticks pass between loops by being admitted as facts at those crossings.

A tick is a fact of resolution. It does not contain the facts it resolved over; it references them by identity. Those referenced facts may themselves be ticks. Levels are therefore not structural tiers; they are recoverable traces of prior resolutions.

Weight is reference density, not authority. A heavily referenced fact is not more true; it has simply been taken up more often by observers, citations, and derivations.
That feels much closer to your actual model.
My updated “where it breaks” answer
It breaks when a user or implementation forgets the split:
facts record concern; vertices govern consequence.
Everything else follows from that.
If someone treats facts as truth, it breaks.

If someone treats weight as authority, it breaks.

If someone lets Horizon/Potential become facts without an explicit topology/authority decision, it breaks.

If references are too weak to recover resolution trails, it breaks.

If observer identity is too vague to make “cared to record it” meaningful, it breaks.

But with your clarified constraints, the model is not especially unsafe. It is permissive at the record layer and strict at the operational crossing layer. That is a pretty good architecture.

Yes — Fact may be the wrong word precisely because it sounds like it smuggles in truth.
You have been using it in the older event-sourcing-ish sense: “a thing that happened / was recorded.” But your model is now sharper than that:

an observer cared to emit an attributed record into fact-space.
That is not a “fact” in ordinary speech. It is closer to a notice, claim, mark, record, or trace.
The problem with “Fact”
“Fact” has good properties:
short
familiar in event-sourcing circles
implies immutability
works well with “fact-space”
But it has one fatal everyday implication:
fact = true statement
Your model explicitly does not mean that. So every doc will have to keep saying “facts are not truth,” which is a smell.
The more you lean into observers, horizon, potential, and “what someone cared to record,” the more Fact feels too epistemically strong.

Best replacements
1. Notice
This may be the strongest semantic fit.
A notice is something an observer noticed enough to emit.
It preserves observer-position naturally. It does not imply truth. It implies attribution, attention, and admission.
Then:

An observation is a notice whose weight begins at one.
But “notice” and “observation” overlap heavily.
Possible canonical:

Notice = base unit
Observation = ingress notice
Tick = derived notice
That is clean.
A tick becomes a derived notice, not a derived fact.

Keeper line:

A notice is not truth; it is attributed concern.
Very good.
Downside: “notice” can sound like a notification or announcement.

2. Mark
Very strong, especially with references.
An observer leaves a mark.
A mark does not claim truth. It is a trace left by someone/something. It works with goblin toothmarks too.
Mark-space
Observation = ingress mark

“Mark” is wonderfully primitive.
Keeper line:

A mark means something left a trace, not that the trace is true.
Downside: “mark-space” is less natural than fact-space. Also “mark” can mean target/victim or grade.
3. Trace
This is excellent conceptually.
A trace is evidence that something passed through.
It fits your semantic trace instincts. It is attribution-friendly, not truth-claiming, and good for provenance.
Trace-space
Observation = ingress trace
Tick = derived trace / resolution trace
Reference = trace relation
Keeper line:
A trace records passage, not truth.
Downside: “trace” is already overloaded in observability and distributed tracing. That may be good or bad.

6. Record
Boring and good.

7. Sign
Interesting, especially semiotic.

8. Token
Mechanically good, but overloaded with LLMs/auth/parser tokens.

9. Datum
Technically correct-ish: a given.
Downside: dead and boring.

10. Event
Probably wrong.
Your unit is not necessarily an event; it can be a claim about a fact, a reference, a derivation, a memory, a relationship, a configuration, etc.

My favorite: Notice
I think Notice may be the best replacement if you want the observer-centric model.
Because it says:

something was noticed, by someone/something, under some horizon, and emitted.
Then the core becomes:
Notice-space admits notices.

An observation is an ingress notice whose weight begins at one.

A tick is a notice produced by resolution, referencing the notices it resolved over.

Weight is notice-density: how often notices are noticed by later notices.
That last line is surprisingly strong:
Weight is how often a notice has been noticed.
This preserves your “amount of times observers have emitted facts about facts” idea beautifully.
A tick is no longer “a weighted fact.” It is:

a derived notice with references.
Or:
a resolution notice.
